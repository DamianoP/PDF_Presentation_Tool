# Server Overview

The backend of the PDF Presentation Tool is a minimalist Node.js application (`server.js`) that leverages the Express framework and the `ws` package. Its primary roles are to serve the static frontend files, facilitate real-time WebSocket communication for remote control and audience interactions, and generate QR codes dynamically.

## 1. Module Purpose and Architecture

The `server.js` module is the entry point for the backend. It does not handle any application business logic related to PDF parsing, rendering, or presentation state management. Instead, it acts as a lightweight infrastructure layer.

### Dependencies

*   `express`: A fast, unopinionated, minimalist web framework for Node.js used for routing and serving static files.
*   `http`: A core Node.js module used to create the HTTP server instance that Express and the WebSocket server bind to.
*   `ws`: A simple to use, blazing fast, and thoroughly tested WebSocket client and server implementation.
*   `qrcode`: A Node.js library for generating QR codes.
*   `path` & `fs`: Core Node.js modules for file system operations and path manipulation.

## 2. Execution Flow and Components

The server initializes by reading connection data, setting up Express middleware for analytics and static file delivery, establishing a REST endpoint for QR codes, and finally mounting a WebSocket server on the HTTP instance.

### 2.1. Analytics and Visit Counter

The server maintains a simple visit counter to track unique usage. It reads from and writes to a text file (`/connectionsCounter.txt`).

*   **Initialization:** Upon startup, the server attempts to read `/connectionsCounter.txt`. If the file exists, it parses the content as an integer to restore the counter. If the file is missing, it creates the file and initializes the counter to `0`.
*   **Tracking Middleware:** An Express middleware intercepts requests to the root path (`/`) or any HTML file (`.html`). It parses cookies to check for the presence of a `pdf_presenter_visited` cookie.
*   **Increment Logic:** If the cookie is absent, it signifies a new unique visit. The server increments `totalConnections`, sets a persistent cookie (`Max-Age=86400`, 1 day) to prevent duplicate counting in the short term, and asynchronously writes the updated count to `/connectionsCounter.txt`.

### 2.2. Static File Serving

The Express application is configured to serve static assets from the `/dist` directory.

*   **Caching Strategy:** A custom middleware is applied to requests for HTML files (`/` and `*.html`). It sets HTTP headers (`Cache-Control: no-cache, no-store, must-revalidate`, `Pragma: no-cache`, `Expires: 0`) to enforce a network-first strategy, ensuring clients always receive the latest version of the application interface.
*   **Asset Delivery:** The `express.static` middleware handles requests for JavaScript bundles, CSS stylesheets, images, and other static resources located in the build output directory (`/dist`).

### 2.3. QR Code Generation API

The server exposes a REST endpoint for generating QR codes, which are used to pair remote controls and launch audience Q&A sessions.

*   **Endpoint:** `GET /api/qr`
*   **Input Parameters:** Requires a query parameter `url` containing the data to encode.
*   **Execution:** The endpoint extracts the `url` parameter. If missing, it responds with a `400 Bad Request`. If present, it utilizes the `qrcode` library to generate a PNG buffer of the QR code with specific styling (colors, margins, dimensions).
*   **Output:** Returns the generated PNG image buffer with the `Content-Type: image/png` header and cache control directives.

## 3. WebSocket Relay System

The core feature of the server is its WebSocket implementation, which enables bi-directional communication between the presenter, remote controls, and audience members.

### 3.1. Session Management

The server maintains a `Map` named `sessions` to track active presentation sessions. Each session is identified by a unique `sessionId` (a string, typically a 64-character UUID generated by the client) and contains:

*   `presenter`: A reference to the WebSocket connection of the primary presenter window.
*   `remotes`: A `Set` of WebSocket connections for active remote control devices (phones).
*   `qaClients`: A `Set` of WebSocket connections for audience members participating in Q&A.

### 3.2. Message Handling

The WebSocket server listens for connections on the `/ws` path. When a client connects, the server assigns a role (`presenter`, `remote`, or `qa-audience`) and a `sessionId` based on the messages it receives.

#### Client Actions

1.  **`create-session` (Presenter):**
    *   **Input:** `{ type: 'create-session', sessionId: '...' }`
    *   **Logic:** The server registers the connection as a `presenter`. It checks if a session with the given ID already exists. If so, it cleans up old connections to prevent conflicts. It creates a new entry in the `sessions` map and acknowledges the creation to the presenter.
2.  **`join-session` (Remote):**
    *   **Input:** `{ type: 'join-session', sessionId: '...' }`
    *   **Logic:** The server attempts to find the corresponding session. If the session or the presenter's connection is invalid, it sends an error and closes the connection. Otherwise, it adds the remote connection to the session's `remotes` set, acknowledges the join, and notifies the presenter of the new remote count.
3.  **`join-qa` (Audience):**
    *   **Input:** `{ type: 'join-qa', sessionId: '...' }`
    *   **Logic:** Similar to `join-session`, but for audience members. It verifies the session and adds the connection to the `qaClients` set.
4.  **`qa-submit` (Audience):**
    *   **Input:** `{ type: 'qa-submit', text: '...' }`
    *   **Logic:** Validates the sender's role. If valid, it forwards the question text (truncated to 500 characters) and a timestamp to the presenter's WebSocket connection.
5.  **`remote-command` (Remote):**
    *   **Input:** `{ type: 'remote-command', command: 'next' | 'prev' }`
    *   **Logic:** Validates the sender's role. If valid, it forwards the command payload directly to the presenter's WebSocket connection to execute slide navigation.
6.  **`state-update` (Presenter):**
    *   **Input:** `{ type: 'state-update', currentPage: 5, totalPages: 10 }`
    *   **Logic:** The presenter broadcasts state changes. The server iterates through the `remotes` set for the session and relays the state update, allowing remote controls to display the current slide number.

### 3.3. Connection Lifecycle and Heartbeat

The server robustly handles disconnects and dead connections.

*   **Heartbeat Mechanism:** A `setInterval` routine runs every 30 seconds. It iterates through all active WebSocket clients. It checks an `isAlive` flag. If false, the connection is terminated. Otherwise, it sets the flag to false and sends a ping. Clients responding with a pong reset the `isAlive` flag to true.
*   **Disconnect Handling (`close` event):**
    *   If a `presenter` disconnects, the server notifies all associated `remotes` and `qaClients` that the session is closed, gracefully terminates their connections, and deletes the session from the map.
    *   If a `remote` disconnects, the server removes it from the `remotes` set and notifies the presenter of the updated remote count.
    *   If a `qa-audience` client disconnects, it is simply removed from the `qaClients` set.
